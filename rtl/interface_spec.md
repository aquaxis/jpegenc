## 1. 概要

本書は、Baseline JPEG Encoder の FPGA 実装におけるモジュール間インターフェース仕様を定義する。
エンコーダは以下の 8 モジュールで構成されるパイプラインアーキテクチャを採用し、
すべてのモジュール間接続に AXI4-Stream プロトコルを使用する。

### 1.1 モジュール一覧

| No. | モジュール名         | 機能                                      |
|-----|----------------------|-------------------------------------------|
| 1   | `rgb2ycbcr`          | 色空間変換 (RGB → YCbCr, BT.601)          |
| 2   | `dct_2d`             | 2次元離散コサイン変換 (8x8 ブロック)       |
| 3   | `quantizer`          | DCT 係数量子化                             |
| 4   | `zigzag_scan`        | ジグザグスキャン並び替え                   |
| 5   | `rle_encoder`        | ランレングス符号化                         |
| 6   | `huffman_encoder`    | ハフマン符号化                             |
| 7   | `bitstream_assembler`| JPEG ビットストリーム組立                  |
| 8   | `jpeg_encoder_top`   | トップレベルモジュール                     |

### 1.2 基本仕様

- **プロトコル**: AXI4-Stream (全モジュール間インターフェース)
- **共通クロック**: `aclk` (単一クロックドメイン)
- **共通リセット**: `aresetn` (Active-Low 非同期リセット)
- **色空間**: 4:4:4 (クロマサブサンプリングなし)
- **JPEG プロファイル**: Baseline JPEG (8-bit DCT, Sequential)

## 3. 共通信号定義

すべてのモジュール間インターフェースは AXI4-Stream プロトコルに準拠する。
以下の信号がすべてのインターフェースで共通的に使用される。

| 信号名     | 方向             | 幅       | 説明                                                    |
|------------|------------------|----------|---------------------------------------------------------|
| `aclk`     | Input            | 1        | システムクロック                                        |
| `aresetn`  | Input            | 1        | Active-Low 非同期リセット                               |
| `tdata`    | Master → Slave   | 可変     | データペイロード (各ステージで幅が異なる)                |
| `tvalid`   | Master → Slave   | 1        | データ有効信号 (Master がデータを出力中であることを示す) |
| `tready`   | Slave → Master   | 1        | Slave 受信準備完了信号 (バックプレッシャ)               |
| `tlast`    | Master → Slave   | 1        | パケット内最終転送を示す                                |
| `tuser`    | Master → Slave   | 可変     | サイドバンドユーザ信号 (各ステージで幅が異なる)         |
| `tkeep`    | Master → Slave   | 可変     | バイト修飾子 (有効バイト指定)                           |

### 3.1 AXI4-Stream ハンドシェイクルール

- データ転送は `tvalid` と `tready` がともに High のクロックエッジで成立する。
- Master は `tready` が Low でも `tvalid` をアサートしてよい。
- Master は `tvalid` をアサートした後、転送が成立するまで `tvalid` と `tdata` を保持しなければならない。
- Slave は `tvalid` に依存せず `tready` をアサートしてよい。

### 4.1 jpeg_encoder_top (トップレベルモジュール)

#### 4.1.1 機能概要

エンコーダ全体のトップレベルモジュール。内部に全パイプラインモジュールをインスタンスし、
外部との入出力インターフェースおよびコンフィギュレーションレジスタを提供する。

#### 4.1.2 ポート一覧

| ポート名             | 方向   | 幅     | 説明                                              |
|----------------------|--------|--------|---------------------------------------------------|
| `aclk`               | Input  | 1      | システムクロック                                  |
| `aresetn`            | Input  | 1      | Active-Low 非同期リセット                         |
| **Slave AXI4-Stream 入力 (ピクセル入力)**                                              |
| `s_axis_tdata`       | Input  | [31:0] | A8R8G8B8 ピクセルデータ                           |
| `s_axis_tvalid`      | Input  | 1      | 入力データ有効                                    |
| `s_axis_tready`      | Output | 1      | 入力受信準備完了                                  |
| `s_axis_tlast`       | Input  | 1      | 行末 (End of Row)                                 |
| `s_axis_tuser`       | Input  | [1:0]  | {EOF[1], SOF[0]}                                  |
| `s_axis_tkeep`       | Input  | [3:0]  | バイトイネーブル                                  |
| **Master AXI4-Stream 出力 (JPEG バイトストリーム)**                                    |
| `m_axis_tdata`       | Output | [7:0]  | JPEG バイトストリームデータ                       |
| `m_axis_tvalid`      | Output | 1      | 出力データ有効                                    |
| `m_axis_tready`      | Input  | 1      | 出力先受信準備完了                                |
| `m_axis_tlast`       | Output | 1      | JPEG データ末尾 (End of JPEG Data)                |
| `m_axis_tuser`       | Output | [1:0]  | {EOF[1], SOF[0]}                                  |
| `m_axis_tkeep`       | Output | [0:0]  | バイト有効                                        |
| **コンフィギュレーションレジスタ**                                                     |
| `img_width`          | Input  | [11:0] | 画像幅 (ピクセル単位, 1-4096)                     |
| `img_height`         | Input  | [11:0] | 画像高さ (ピクセル単位, 1-4096)                   |
| `img_width_valid`    | Input  | 1      | 画像幅ロードパルス                                |
| `img_height_valid`   | Input  | 1      | 画像高さロードパルス                              |
| `qp_level`           | Input  | [3:0]  | 量子化パラメータ (0=最高品質, 15=最小サイズ)      |

#### 4.1.3 s_axis_tdata フォーマット (入力ピクセル)

```
  31      24 23      16 15       8 7        0
 +----------+----------+----------+----------+
 | A[7:0]   | R[7:0]   | G[7:0]   | B[7:0]   |
 +----------+----------+----------+----------+
   Alpha      Red        Green      Blue
   (未使用)
```

- Alpha チャネルは無視される (将来的な拡張用に予約)。
- 各色成分は符号なし 8-bit (0-255)。

#### 4.1.4 s_axis_tuser フォーマット

```
  1       0
 +--------+--------+
 | EOF    | SOF    |
 +--------+--------+
```

| ビット | 名称 | 説明                                             |
|--------|------|--------------------------------------------------|
| [0]    | SOF  | Start of Frame: フレーム最初のピクセルで 1       |
| [1]    | EOF  | End of Frame: フレーム最後のピクセルで 1         |

#### 4.1.5 tlast セマンティクス

- **入力側 (`s_axis_tlast`)**: 各ラスター行の最後のピクセルで High。
- **出力側 (`m_axis_tlast`)**: JPEG ファイル全体の最後のバイト (EOI マーカーの末尾) で High。

#### 4.1.6 コンフィギュレーションレジスタ詳細

- `img_width` / `img_height`: フレーム処理開始前に `img_width_valid` / `img_height_valid` パルスとともに設定する。フレーム処理中の変更は禁止。
- `qp_level`: 量子化テーブルのスケーリングファクタ。将来使用のためのオプション入力。初期実装ではデフォルトの JPEG 標準量子化テーブルを使用する。

### 4.2 rgb2ycbcr (色空間変換)

#### 4.2.1 機能概要

RGB 色空間から YCbCr 色空間への変換を行う。ITU-R BT.601 規格の変換係数を使用する。
固定小数点演算パイプラインにより 3-4 クロックサイクルの遅延で変換を実行する。

#### 4.2.2 ポート一覧

| ポート名             | 方向   | 幅     | 説明                                              |
|----------------------|--------|--------|---------------------------------------------------|
| `aclk`               | Input  | 1      | システムクロック                                  |
| `aresetn`            | Input  | 1      | Active-Low 非同期リセット                         |
| **Slave AXI4-Stream 入力 (RGB ピクセル)**                                              |
| `s_axis_tdata`       | Input  | [31:0] | A8R8G8B8 ピクセルデータ                           |
| `s_axis_tvalid`      | Input  | 1      | 入力データ有効                                    |
| `s_axis_tready`      | Output | 1      | 入力受信準備完了                                  |
| `s_axis_tlast`       | Input  | 1      | 行末 (End of Row)                                 |
| `s_axis_tuser`       | Input  | [1:0]  | {EOF[1], SOF[0]}                                  |
| `s_axis_tkeep`       | Input  | [3:0]  | バイトイネーブル                                  |
| **Master AXI4-Stream 出力 (YCbCr ピクセル)**                                           |
| `m_axis_tdata`       | Output | [23:0] | {Y[23:16], Cb[15:8], Cr[7:0]}                    |
| `m_axis_tvalid`      | Output | 1      | 出力データ有効                                    |
| `m_axis_tready`      | Input  | 1      | 出力先受信準備完了                                |
| `m_axis_tlast`       | Output | 1      | 行末 (End of Row)                                 |
| `m_axis_tuser`       | Output | [1:0]  | {EOF[1], SOF[0]}                                  |

#### 4.2.3 m_axis_tdata フォーマット (出力 YCbCr ピクセル)

```
  23      16 15       8 7        0
 +----------+----------+----------+
 | Y[7:0]   | Cb[7:0]  | Cr[7:0]  |
 +----------+----------+----------+
   Luminance  Chroma-Bl  Chroma-Rd
```

- 各成分は符号なし 8-bit (0-255)。
- Y, Cb, Cr はすべてフルレンジ (JFIF 仕様に準拠)。

#### 4.2.4 tuser フォーマット

入力の `s_axis_tuser` をパイプライン遅延に合わせてそのまま出力側に伝搬する。

```
  1       0
 +--------+--------+
 | EOF    | SOF    |
 +--------+--------+
```

#### 4.2.5 tlast セマンティクス

- 入出力とも行末 (End of Row) を示す。
- パイプライン遅延分だけ遅れて出力される。

#### 4.2.6 変換式 (BT.601)

```
  Y  =  0.299 * R + 0.587 * G + 0.114 * B
  Cb = -0.169 * R - 0.331 * G + 0.500 * B + 128
  Cr =  0.500 * R - 0.419 * G - 0.081 * B + 128
```

#### 4.2.7 固定小数点実装

- 推奨フォーマット: 8.8 固定小数点 (16-bit) または 10.6 固定小数点 (16-bit)。
- 乗算は DSP スライスを活用した multiply-accumulate (MAC) 演算で実装する。
- 各色成分の演算は並列に実施可能。
- 最終結果は飽和演算 (saturation) によりクランプする: 0-255 の範囲。

#### 4.2.8 レイテンシ

- **パイプラインレイテンシ**: 3-4 クロックサイクル
  - Stage 1: 入力ラッチおよび係数乗算
  - Stage 2: 加算/アキュムレート
  - Stage 3: オフセット加算および飽和クランプ
  - Stage 4 (オプション): 出力レジスタ

### 4.3 dct_2d (2次元離散コサイン変換)

#### 4.3.1 機能概要

8x8 ブロック単位の 2 次元 DCT を実行する。入力ピクセルのラスタスキャンストリームを
8 行分バッファリングし、Y/Cb/Cr 各成分を分離して個別に DCT を計算する。
分離可能 (separable) 特性を利用し、行方向 1D-DCT と列方向 1D-DCT を順次実行する。

#### 4.3.2 ポート一覧

| ポート名             | 方向   | 幅     | 説明                                              |
|----------------------|--------|--------|---------------------------------------------------|
| `aclk`               | Input  | 1      | システムクロック                                  |
| `aresetn`            | Input  | 1      | Active-Low 非同期リセット                         |
| **Slave AXI4-Stream 入力 (YCbCr ピクセル)**                                            |
| `s_axis_tdata`       | Input  | [23:0] | {Y[23:16], Cb[15:8], Cr[7:0]}                    |
| `s_axis_tvalid`      | Input  | 1      | 入力データ有効                                    |
| `s_axis_tready`      | Output | 1      | 入力受信準備完了                                  |
| `s_axis_tlast`       | Input  | 1      | 行末 (End of Row)                                 |
| `s_axis_tuser`       | Input  | [1:0]  | {EOF[1], SOF[0]}                                  |
| **Master AXI4-Stream 出力 (DCT 係数)**                                                 |
| `m_axis_tdata`       | Output | [11:0] | 符号付き DCT 係数 (1 クロックにつき 1 係数)       |
| `m_axis_tvalid`      | Output | 1      | 出力データ有効                                    |
| `m_axis_tready`      | Input  | 1      | 出力先受信準備完了                                |
| `m_axis_tlast`       | Output | 1      | ブロック末尾 (係数 index 63)                      |
| `m_axis_tuser`       | Output | [3:0]  | {EOF[3], SOF[2], comp_id[1:0]}                    |
| **コンフィギュレーション入力**                                                         |
| `img_width`          | Input  | [11:0] | 画像幅 (ピクセル単位)                             |
| `img_height`         | Input  | [11:0] | 画像高さ (ピクセル単位)                           |

#### 4.3.3 m_axis_tdata フォーマット (DCT 係数)

```
  11                              0
 +----------------------------------+
 | signed DCT coefficient [11:0]   |
 +----------------------------------+
```

- 12-bit 符号付き整数 (-2048 ~ +2047)。
- 8-bit DCT 入力、8 点 DCT の結果として十分な精度を確保する。

#### 4.3.4 m_axis_tuser フォーマット

```
  3       2       1       0
 +--------+--------+--------+--------+
 | EOF    | SOF    | comp_id[1]      | comp_id[0] |
 +--------+--------+-----------------+------------+
```

| ビット   | 名称         | 説明                                           |
|----------|--------------|------------------------------------------------|
| [1:0]    | `comp_id`    | コンポーネント ID: 00=Y, 01=Cb, 10=Cr          |
| [2]      | `SOF`        | Start of Frame (フレーム最初のブロック)         |
| [3]      | `EOF`        | End of Frame (フレーム最後のブロック)           |

#### 4.3.5 tlast セマンティクス

- 8x8 ブロックの最後の DCT 係数 (index 63) で High。
- 各ブロックの 64 番目の係数出力時にアサートされる。

#### 4.3.6 内部動作

1. **ピクセルバッファリング**: 8 ラスター行分のピクセルデータを内部ラインバッファに蓄積する。
2. **成分分離**: {Y, Cb, Cr} の 3 成分を分離し、個別のバッファに格納する。
3. **レベルシフト**: 符号なし 8-bit 値から符号付きに変換するため -128 のオフセットを適用する。
4. **行方向 1D-DCT**: 8x8 ブロックの各行 (8 サンプル) に対して 1D-DCT を実行する。
5. **転置バッファ**: 行方向 DCT 結果をバッファに格納し、列方向アクセスに備える。
6. **列方向 1D-DCT**: 転置されたデータの各列に対して 1D-DCT を実行する。
7. **出力**: 64 係数をブロック内ラスタオーダーで出力する (行 0 左→右, 行 1 左→右, ..., 行 7 左→右)。

#### 4.3.7 ブロック処理順序

MCU (Minimum Coded Unit) 内のブロック出力順序:

```
  1. Y  ブロック (comp_id = 00)
  2. Cb ブロック (comp_id = 01)
  3. Cr ブロック (comp_id = 10)
```

4:4:4 サブサンプリングのため、各 MCU は 1Y + 1Cb + 1Cr = 3 ブロックで構成される。

#### 4.3.8 レイテンシ

- **初期レイテンシ**: 8 行分のピクセルバッファリング + DCT 演算パイプライン遅延。
  - 画像幅 W のとき、最低 8W クロックサイクルのバッファリング遅延。
- **ブロック間スループット**: 64 クロックサイクル/ブロック (1 係数/クロック出力時)。
- **DCT 演算パイプライン**: 1D-DCT あたり 8-16 クロックサイクル (実装依存)。

### 4.4 quantizer (量子化)

#### 4.4.1 機能概要

DCT 係数を JPEG 標準量子化テーブルで除算 (量子化) する。
輝度 (Luminance) と色差 (Chrominance) で異なる量子化テーブルを使用し、
`comp_id` に基づいてテーブルを切り替える。
除算はハードウェア効率のため逆数乗算で実装する。

#### 4.4.2 ポート一覧

| ポート名             | 方向   | 幅     | 説明                                              |
|----------------------|--------|--------|---------------------------------------------------|
| `aclk`               | Input  | 1      | システムクロック                                  |
| `aresetn`            | Input  | 1      | Active-Low 非同期リセット                         |
| **Slave AXI4-Stream 入力 (DCT 係数)**                                                  |
| `s_axis_tdata`       | Input  | [11:0] | 符号付き DCT 係数                                 |
| `s_axis_tvalid`      | Input  | 1      | 入力データ有効                                    |
| `s_axis_tready`      | Output | 1      | 入力受信準備完了                                  |
| `s_axis_tlast`       | Input  | 1      | ブロック末尾 (係数 index 63)                      |
| `s_axis_tuser`       | Input  | [3:0]  | {EOF[3], SOF[2], comp_id[1:0]}                    |
| **Master AXI4-Stream 出力 (量子化係数)**                                               |
| `m_axis_tdata`       | Output | [11:0] | 符号付き量子化係数                                |
| `m_axis_tvalid`      | Output | 1      | 出力データ有効                                    |
| `m_axis_tready`      | Input  | 1      | 出力先受信準備完了                                |
| `m_axis_tlast`       | Output | 1      | ブロック末尾 (係数 index 63)                      |
| `m_axis_tuser`       | Output | [3:0]  | {EOF[3], SOF[2], comp_id[1:0]}                    |

#### 4.4.3 tdata フォーマット

入力・出力とも同一フォーマット:

```
  11                              0
 +----------------------------------+
 | signed coefficient [11:0]       |
 +----------------------------------+
```

- 入力: DCT 係数 (符号付き 12-bit)
- 出力: 量子化済み係数 (符号付き 12-bit、ただし量子化後は値域が縮小)

#### 4.4.4 tuser フォーマット

入力と同一フォーマットをパイプライン遅延に合わせて伝搬する。

```
  3       2       1       0
 +--------+--------+--------+--------+
 | EOF    | SOF    | comp_id[1:0]    |
 +--------+--------+-----------------+
```

#### 4.4.5 tlast セマンティクス

- ブロックの最後の量子化係数 (index 63) で High。
- 入力の `tlast` をパイプライン遅延に合わせて伝搬する。

#### 4.4.6 内部動作

1. **量子化テーブル選択**: `comp_id` に基づいて量子化テーブルを選択する。
   - `comp_id == 2'b00` (Y): 輝度量子化テーブル (Luminance Q-table)
   - `comp_id == 2'b01` (Cb) または `comp_id == 2'b10` (Cr): 色差量子化テーブル (Chrominance Q-table)
2. **係数インデックス管理**: 内部カウンタ (0-63) で現在の係数位置を追跡する。各ブロックの先頭でリセットする。
3. **量子化演算**: `Q_coeff = round(DCT_coeff / Q_table[index])`
4. **逆数乗算**: 除算を避けるため、事前計算した逆数テーブルとの乗算で実装する。
   - `Q_coeff = (DCT_coeff * reciprocal[index] + rounding_offset) >> shift_amount`
5. **丸め処理**: 0 方向への丸め (truncation toward zero) を使用する。

#### 4.4.7 レイテンシ

- **パイプラインレイテンシ**: 2-3 クロックサイクル
  - Stage 1: テーブルルックアップおよび逆数乗算
  - Stage 2: シフトおよび丸め処理
  - Stage 3 (オプション): 出力レジスタ

### 4.5 zigzag_scan (ジグザグスキャン)

#### 4.5.1 機能概要

8x8 ブロック内の量子化済み DCT 係数をラスタオーダーからジグザグオーダーに並び替える。
64 係数分のブロックバッファリングを行い、ジグザグ順で読み出す。

#### 4.5.2 ポート一覧

| ポート名             | 方向   | 幅     | 説明                                              |
|----------------------|--------|--------|---------------------------------------------------|
| `aclk`               | Input  | 1      | システムクロック                                  |
| `aresetn`            | Input  | 1      | Active-Low 非同期リセット                         |
| **Slave AXI4-Stream 入力 (ラスタオーダー係数)**                                        |
| `s_axis_tdata`       | Input  | [11:0] | 符号付き量子化係数 (ラスタオーダー)               |
| `s_axis_tvalid`      | Input  | 1      | 入力データ有効                                    |
| `s_axis_tready`      | Output | 1      | 入力受信準備完了                                  |
| `s_axis_tlast`       | Input  | 1      | ブロック末尾                                      |
| `s_axis_tuser`       | Input  | [3:0]  | {EOF[3], SOF[2], comp_id[1:0]}                    |
| **Master AXI4-Stream 出力 (ジグザグオーダー係数)**                                     |
| `m_axis_tdata`       | Output | [11:0] | 符号付き量子化係数 (ジグザグオーダー)             |
| `m_axis_tvalid`      | Output | 1      | 出力データ有効                                    |
| `m_axis_tready`      | Input  | 1      | 出力先受信準備完了                                |
| `m_axis_tlast`       | Output | 1      | ブロック末尾                                      |
| `m_axis_tuser`       | Output | [3:0]  | {EOF[3], SOF[2], comp_id[1:0]}                    |

#### 4.5.3 tdata フォーマット

入力・出力とも同一ビット幅:

```
  11                              0
 +----------------------------------+
 | signed quantized coeff [11:0]   |
 +----------------------------------+
```

- 入力: ラスタオーダー (行 0 左→右, 行 1 左→右, ..., 行 7 左→右)
- 出力: ジグザグオーダー (対角線走査順)

#### 4.5.4 tuser フォーマット

入力の `tuser` を伝搬する。ブロック単位で保持し、出力時に付与する。

```
  3       2       1       0
 +--------+--------+--------+--------+
 | EOF    | SOF    | comp_id[1:0]    |
 +--------+--------+-----------------+
```

#### 4.5.5 tlast セマンティクス

- ジグザグオーダーの最後の係数 (ジグザグ index 63) で High。

#### 4.5.6 内部動作

1. **ブロックバッファリング**: 64 個の係数 (1 ブロック分) を内部 RAM/レジスタに書き込む。
2. **アドレス変換**: ジグザグオーダーのルックアップテーブル (LUT) に従って読み出しアドレスを生成する。
3. **読み出し**: ジグザグ順で係数を読み出し、出力ストリームに送出する。

ジグザグスキャンオーダー (JPEG 標準):

```
  ラスタ index →  ジグザグ出力順序

   0  1  5  6 14 15 27 28
   2  4  7 13 16 26 29 42
   3  8 12 17 25 30 41 43
   9 11 18 24 31 40 44 53
  10 19 23 32 39 45 52 54
  20 22 33 38 46 51 55 60
  21 34 37 47 50 56 59 61
  35 36 48 49 57 58 62 63
```

#### 4.5.7 レイテンシ

- **ブロックバッファリング遅延**: 64 クロックサイクル以上 (1 ブロック全体の書き込み完了後に読み出し開始)。
- ダブルバッファリング実装により、1 ブロック書き込み中に前ブロックの読み出しを並行実行可能。

### 4.6 rle_encoder (ランレングス符号化)

#### 4.6.1 機能概要

ジグザグ順に並んだ量子化係数に対してランレングス符号化を行う。
DC 係数には DPCM (差分パルス符号変調) を適用し、AC 係数には (zero_run, size, amplitude)
形式のシンボルを生成する。

#### 4.6.2 ポート一覧

| ポート名             | 方向   | 幅     | 説明                                              |
|----------------------|--------|--------|---------------------------------------------------|
| `aclk`               | Input  | 1      | システムクロック                                  |
| `aresetn`            | Input  | 1      | Active-Low 非同期リセット                         |
| **Slave AXI4-Stream 入力 (ジグザグオーダー係数)**                                      |
| `s_axis_tdata`       | Input  | [11:0] | 符号付き量子化係数 (ジグザグオーダー)             |
| `s_axis_tvalid`      | Input  | 1      | 入力データ有効                                    |
| `s_axis_tready`      | Output | 1      | 入力受信準備完了                                  |
| `s_axis_tlast`       | Input  | 1      | ブロック末尾                                      |
| `s_axis_tuser`       | Input  | [3:0]  | {EOF[3], SOF[2], comp_id[1:0]}                    |
| **Master AXI4-Stream 出力 (RLE シンボル)**                                             |
| `m_axis_tdata`       | Output | [23:0] | RLE シンボル (`rle_symbol_t`)                     |
| `m_axis_tvalid`      | Output | 1      | 出力データ有効                                    |
| `m_axis_tready`      | Input  | 1      | 出力先受信準備完了                                |
| `m_axis_tlast`       | Output | 1      | ブロック末尾 (EOB または最後の AC シンボル)       |
| `m_axis_tuser`       | Output | [3:0]  | {EOF[3], SOF[2], comp_id[1:0]}                    |

#### 4.6.3 m_axis_tdata フォーマット (rle_symbol_t)

```
  23    20 19    16 15                    4 3      2      1      0
 +--------+--------+----------------------+--------+------+------+
 |zero_run| size   | amplitude[11:0]      | is_dc  |is_eob| rsvd |
 +--------+--------+----------------------+--------+------+------+
   4-bit    4-bit    12-bit (signed)        1-bit   1-bit  2-bit
```

| フィールド      | ビット    | 幅   | 説明                                             |
|-----------------|-----------|------|--------------------------------------------------|
| `zero_run`      | [23:20]   | 4    | 先行ゼロラン長 (0-15)                            |
| `size`          | [19:16]   | 4    | カテゴリ/サイズ (0-11)                           |
| `amplitude`     | [15:4]    | 12   | 係数振幅 (符号付き)                              |
| `is_dc`         | [3]       | 1    | 1 = DC 係数, 0 = AC 係数                         |
| `is_eob`        | [2]       | 1    | 1 = End of Block                                 |
| `reserved`      | [1:0]     | 2    | 予約 (0 固定)                                    |

#### 4.6.4 tuser フォーマット

入力の `tuser` を伝搬する。

```
  3       2       1       0
 +--------+--------+--------+--------+
 | EOF    | SOF    | comp_id[1:0]    |
 +--------+--------+-----------------+
```

#### 4.6.5 tlast セマンティクス

- ブロックの最後の RLE シンボルで High。
- EOB シンボルが出力される場合は EOB が `tlast`。
- ブロック内のすべての AC 係数が非ゼロの場合は、最後の AC シンボルが `tlast`。

#### 4.6.6 内部動作

**DC 係数処理 (ジグザグ index 0):**

1. 同一コンポーネントの前ブロック DC 値との差分を計算する (DPCM)。
   - `DC_diff = DC_current - DC_previous`
2. 差分値のカテゴリ (size) を算出する。
3. シンボルを出力する: `zero_run=0, size=category, amplitude=DC_diff, is_dc=1, is_eob=0`。
4. 前回 DC 値レジスタを更新する。

**AC 係数処理 (ジグザグ index 1-63):**

1. ゼロ係数を検出し、連続ゼロカウンタをインクリメントする。
2. 非ゼロ係数を検出した場合:
   - ゼロランが 15 を超える場合: `(zero_run=15, size=0)` の ZRL (Zero Run Length) シンボルを 16 ゼロごとに出力する。
   - 残りのゼロランと非ゼロ係数のカテゴリ/振幅をシンボルとして出力する。
3. ブロック末尾まですべて処理した後、残りの AC 係数がすべてゼロの場合:
   - EOB シンボルを出力する: `zero_run=0, size=0, is_eob=1`。

**DC 値レジスタ:**

- Y, Cb, Cr の 3 成分分の DC 値レジスタを保持する (計 3 個)。
- SOF (Start of Frame) 受信時にすべてゼロにリセットする。

#### 4.6.7 Size/Category テーブル

| Category (size) | 値の範囲                            |
|-----------------|-------------------------------------|
| 0               | 0                                   |
| 1               | -1, 1                               |
| 2               | -3..-2, 2..3                        |
| 3               | -7..-4, 4..7                        |
| 4               | -15..-8, 8..15                      |
| 5               | -31..-16, 16..31                    |
| 6               | -63..-32, 32..63                    |
| 7               | -127..-64, 64..127                  |
| 8               | -255..-128, 128..255                |
| 9               | -511..-256, 256..511                |
| 10              | -1023..-512, 512..1023              |
| 11              | -2047..-1024, 1024..2047            |

#### 4.6.8 レイテンシ

- **可変レイテンシ**: 入力係数のゼロ分布に依存する。
  - 最良ケース (全非ゼロ): 入力と同期して 1 シンボル/クロック。
  - 最悪ケース (全ゼロ): ブロック全体を消費後に EOB を 1 シンボル出力。
- 出力レートは入力レートと異なる可能性がある (データ依存)。

### 4.7 huffman_encoder (ハフマン符号化)

#### 4.7.1 機能概要

RLE シンボルをハフマン符号に変換する。DC/AC、輝度/色差の 4 種類のハフマンテーブルを使用し、
可変長符号語 (codeword) と追加ビット (additional bits) を結合して出力する。

#### 4.7.2 ポート一覧

| ポート名             | 方向   | 幅     | 説明                                              |
|----------------------|--------|--------|---------------------------------------------------|
| `aclk`               | Input  | 1      | システムクロック                                  |
| `aresetn`            | Input  | 1      | Active-Low 非同期リセット                         |
| **Slave AXI4-Stream 入力 (RLE シンボル)**                                              |
| `s_axis_tdata`       | Input  | [23:0] | RLE シンボル (`rle_symbol_t`)                     |
| `s_axis_tvalid`      | Input  | 1      | 入力データ有効                                    |
| `s_axis_tready`      | Output | 1      | 入力受信準備完了                                  |
| `s_axis_tlast`       | Input  | 1      | ブロック末尾                                      |
| `s_axis_tuser`       | Input  | [3:0]  | {EOF[3], SOF[2], comp_id[1:0]}                    |
| **Master AXI4-Stream 出力 (ハフマン符号語)**                                           |
| `m_axis_tdata`       | Output | [31:0] | MSB アラインド可変長符号語                        |
| `m_axis_tvalid`      | Output | 1      | 出力データ有効                                    |
| `m_axis_tready`      | Input  | 1      | 出力先受信準備完了                                |
| `m_axis_tlast`       | Output | 1      | ブロック末尾 (最後の符号語)                       |
| `m_axis_tuser`       | Output | [6:0]  | {EOF[6], SOF[5], code_length[4:0]}                |

#### 4.7.3 m_axis_tdata フォーマット (ハフマン符号語)

```
  31                                                              0
 +----------------------------------------------------------------+
 | Huffman code bits | Additional bits | Padding (0)              |
 +----------------------------------------------------------------+
   <-- MSB aligned, left-justified -->           <-- unused bits -->
```

- ハフマン符号ビットと追加ビットが MSB 側から連結される。
- 未使用ビット (下位側) は 0 で埋められる。
- 有効ビット数は `tuser` の `code_length` フィールドで示される。

例: ハフマン符号 = `1110` (4 bits), 追加ビット = `011` (3 bits) の場合:

```
  tdata[31:0] = 1110_011_0_0000_0000_0000_0000_0000_0000 (= 0xE600_0000)
  code_length = 7
```

#### 4.7.4 m_axis_tuser フォーマット

```
  6       5       4                   0
 +--------+--------+-------------------+
 | EOF    | SOF    | code_length[4:0]  |
 +--------+--------+-------------------+
```

| フィールド      | ビット    | 幅   | 説明                                             |
|-----------------|-----------|------|--------------------------------------------------|
| `code_length`   | [4:0]     | 5    | 有効ビット数 (2-27)                              |
| `SOF`           | [5]       | 1    | Start of Frame                                   |
| `EOF`           | [6]       | 1    | End of Frame                                     |

- `code_length` の最小値: 2 (最短ハフマン符号)。
- `code_length` の最大値: 27 (最長ハフマン符号 16 bits + 最大追加ビット 11 bits)。

#### 4.7.5 tlast セマンティクス

- ブロックの最後の符号語で High。
- 通常は EOB シンボルのハフマン符号が `tlast`。

#### 4.7.6 内部動作

**ハフマンテーブル (4 テーブル):**

| テーブル名      | 用途                    | comp_id 条件          |
|-----------------|-------------------------|-----------------------|
| `DC_lum`        | DC 輝度ハフマンテーブル | `comp_id == 00` かつ `is_dc == 1` |
| `DC_chrom`      | DC 色差ハフマンテーブル | `comp_id != 00` かつ `is_dc == 1` |
| `AC_lum`        | AC 輝度ハフマンテーブル | `comp_id == 00` かつ `is_dc == 0` |
| `AC_chrom`      | AC 色差ハフマンテーブル | `comp_id != 00` かつ `is_dc == 0` |

**DC シンボル処理:**

1. `size` (category) をキーとして DC ハフマンテーブルをルックアップする。
2. ハフマン符号 + `size` ビットの追加ビット (amplitude の下位ビット) を結合する。
3. 負の振幅の場合: 追加ビットは 1 の補数 (ビット反転) で表現する。

**AC シンボル処理:**

1. `(zero_run, size)` をペアとして AC ハフマンテーブルをルックアップする。
   - ルックアップキー = `(zero_run << 4) | size` (8-bit)。
2. ハフマン符号 + `size` ビットの追加ビットを結合する。
3. 負の振幅の場合: DC と同様に 1 の補数で表現する。

**EOB シンボル処理:**

1. AC ハフマンテーブルでキー `0x00` (zero_run=0, size=0) をルックアップする。
2. ハフマン符号のみを出力する (追加ビットなし)。

**ZRL シンボル処理:**

1. AC ハフマンテーブルでキー `0xF0` (zero_run=15, size=0) をルックアップする。
2. ハフマン符号のみを出力する (追加ビットなし)。

#### 4.7.7 レイテンシ

- **パイプラインレイテンシ**: 1-2 クロックサイクル/シンボル
  - Stage 1: テーブルルックアップおよびビット結合
  - Stage 2 (オプション): 出力レジスタ

### 4.8 bitstream_assembler (ビットストリーム組立)

#### 4.8.1 機能概要

可変長ハフマン符号語をバイトアラインドな JPEG バイトストリームに組み立てる。
ビットアキュムレータ (シフトレジスタ) を用いて可変長データをバイト境界にパッキングし、
バイトスタッフィングおよび JPEG ファイルヘッダ/フッタの生成を行う。

#### 4.8.2 ポート一覧

| ポート名             | 方向   | 幅     | 説明                                              |
|----------------------|--------|--------|---------------------------------------------------|
| `aclk`               | Input  | 1      | システムクロック                                  |
| `aresetn`            | Input  | 1      | Active-Low 非同期リセット                         |
| **Slave AXI4-Stream 入力 (ハフマン符号語)**                                            |
| `s_axis_tdata`       | Input  | [31:0] | MSB アラインド符号語                              |
| `s_axis_tvalid`      | Input  | 1      | 入力データ有効                                    |
| `s_axis_tready`      | Output | 1      | 入力受信準備完了                                  |
| `s_axis_tlast`       | Input  | 1      | ブロック末尾                                      |
| `s_axis_tuser`       | Input  | [6:0]  | {EOF[6], SOF[5], code_length[4:0]}                |
| **Master AXI4-Stream 出力 (JPEG バイトストリーム)**                                    |
| `m_axis_tdata`       | Output | [7:0]  | JPEG バイトストリームデータ                       |
| `m_axis_tvalid`      | Output | 1      | 出力データ有効                                    |
| `m_axis_tready`      | Input  | 1      | 出力先受信準備完了                                |
| `m_axis_tlast`       | Output | 1      | JPEG ファイル末尾                                 |
| `m_axis_tuser`       | Output | [1:0]  | {EOF[1], SOF[0]}                                  |
| `m_axis_tkeep`       | Output | [0:0]  | バイト有効                                        |
| **コンフィギュレーション入力**                                                         |
| `img_width`          | Input  | [11:0] | 画像幅 (ヘッダ生成用)                             |
| `img_height`         | Input  | [11:0] | 画像高さ (ヘッダ生成用)                           |

#### 4.8.3 m_axis_tdata フォーマット

```
  7                0
 +------------------+
 | JPEG byte [7:0]  |
 +------------------+
```

- 1 クロックサイクルにつき 1 バイトを出力する。
- バイトオーダーは JPEG ファイルフォーマット順 (ビッグエンディアン)。

#### 4.8.4 m_axis_tuser フォーマット

```
  1       0
 +--------+--------+
 | EOF    | SOF    |
 +--------+--------+
```

| ビット | 名称 | 説明                                             |
|--------|------|--------------------------------------------------|
| [0]    | SOF  | JPEG ファイルの最初のバイト (SOI マーカーの先頭) |
| [1]    | EOF  | JPEG ファイルの最後のバイト (EOI マーカーの末尾) |

#### 4.8.5 tlast セマンティクス

- JPEG ファイル全体の最後のバイトで High。
- EOI マーカー (`0xFF`, `0xD9`) の 2 バイト目 (`0xD9`) 出力時にアサートされる。

#### 4.8.6 内部動作

**ビットアキュムレータ:**

1. 入力の可変長符号語を内部シフトレジスタ (推奨: 48-64 bit 幅) に追加する。
2. アキュムレータ内に 8 ビット以上蓄積された場合、上位 8 ビットを 1 バイトとして出力する。
3. 複数バイトが蓄積されている場合は連続して出力する。

**バイトスタッフィング:**

- データバイトとして `0xFF` を出力した場合、直後に `0x00` (スタッフバイト) を挿入する。
- これにより JPEG マーカーとデータバイトの誤認を防ぐ。

**SOF (Start of Frame) 処理:**

入力 `tuser` で SOF を検出した場合、エントロピー符号化データの前に以下の JPEG ヘッダマーカーを生成・出力する:

| マーカー  | コード     | 説明                                   |
|-----------|------------|----------------------------------------|
| SOI       | `0xFFD8`   | Start of Image                         |
| APP0/JFIF | `0xFFE0`   | JFIF Application Segment               |
| DQT       | `0xFFDB`   | Define Quantization Table (輝度+色差)  |
| SOF0      | `0xFFC0`   | Start of Frame (Baseline DCT)          |
| DHT       | `0xFFC4`   | Define Huffman Table (DC/AC 各 2 種)   |
| SOS       | `0xFFDA`   | Start of Scan                          |

SOF0 マーカーセグメント内で `img_width` と `img_height` をエンコードする。

**EOF (End of Frame) 処理:**

1. アキュムレータ内の残りビットを 1-bit パディング (すべて 1) でバイト境界まで埋める。
2. パディングされたバイトを出力する (バイトスタッフィングを適用)。
3. EOI マーカー (`0xFFD9`) を出力する。
4. EOI マーカーの最終バイト出力時に `m_axis_tlast` をアサートする。

#### 4.8.7 JPEG ヘッダ構造 (参考)

```
  FFD8                         -- SOI
  FFE0 xxxx ...                -- APP0 (JFIF)
  FFDB xxxx ...                -- DQT (Luminance table)
  FFDB xxxx ...                -- DQT (Chrominance table)
  FFC0 xxxx ...                -- SOF0 (img_height, img_width, 3 components)
  FFC4 xxxx ...                -- DHT (DC Luminance)
  FFC4 xxxx ...                -- DHT (DC Chrominance)
  FFC4 xxxx ...                -- DHT (AC Luminance)
  FFC4 xxxx ...                -- DHT (AC Chrominance)
  FFDA xxxx ...                -- SOS
  [Entropy-coded data]         -- Huffman coded bitstream (with byte stuffing)
  FFD9                         -- EOI
```

#### 4.8.8 レイテンシ

- **ヘッダ出力遅延**: SOF 検出後、ヘッダバイト列の出力完了までに数百クロックサイクルを要する (ヘッダサイズ依存)。
- **ビットパッキング遅延**: 1-2 クロックサイクル (アキュムレータへの追加とバイト出力)。
- **バイトスタッフィング遅延**: `0xFF` データバイト出力後、追加の 1 クロックサイクル (スタッフバイト挿入)。

## 5. タイミング図

### 5.1 基本 AXI4-Stream 転送

通常の valid/ready ハンドシェイクによるデータ転送を示す。
データ転送は `tvalid` と `tready` がともに High のクロック立ち上がりエッジで成立する。

```
             ___     ___     ___     ___     ___     ___     ___     ___
  aclk    __|   |___|   |___|   |___|   |___|   |___|   |___|   |___|   |__
                 ___________________________________
  tvalid  ______|                                   |______________________
                 ___________________________________
  tready  ______|                                   |______________________
                 _________ _________ _________ _________
  tdata   XXXXXX___D0____X___D1____X___D2____X___D3____XXXXXXXXXXXXXXXXXXXXXX
                                               _________
  tlast   ____________________________________|         |______________________

  Transfer:       ^T0      ^T1      ^T2      ^T3
                  (valid & ready 両方 High で転送成立)
```

- T0-T3: データ D0-D3 が転送されるクロックサイクル。
- D3 の転送時に `tlast` がアサートされ、パケット末尾を示す。

### 5.2 バックプレッシャ例

Slave が `tready` をデアサートして転送を一時停止する場合を示す。

```
             ___     ___     ___     ___     ___     ___     ___     ___
  aclk    __|   |___|   |___|   |___|   |___|   |___|   |___|   |___|   |__
                 _______________________________________________
  tvalid  ______|                                               |__________
                 _________                   ___________________
  tready  ______|         |_________________|                   |__________
                 _________                   _________ _________
  tdata   XXXXXX___D0____X________D1________X___D1____X___D2____XXXXXXXXXXX

  Transfer:       ^T0                        ^T1      ^T2
                        ^^^^^^^^^^^^^^^^^^^^
                        tready Low: Master は D1 を保持し続ける
                        (tvalid と tdata を維持する義務がある)
```

- T0: D0 の転送が成立する。
- `tready` がデアサートされると、Master は `tvalid` と `tdata` (D1) を保持し続ける。
- `tready` が再アサートされたサイクルで T1 (D1) の転送が成立する。

### 5.3 SOF/EOF シグナリング

フレーム開始 (SOF) とフレーム終了 (EOF) が `tuser` を通じてパイプラインに伝搬する様子を示す。

```
             ___     ___     ___     ___     ___               ___     ___
  aclk    __|   |___|   |___|   |___|   |___|   |__ ... _____|   |___|   |__
                 _______________________________________________
  tvalid  ______|                                               ... ______
                 _______________________________________________
  tready  ______|                                               ... ______
                 _________ _________ _________ _________       _________
  tdata   XXXXXX___P0____X___P1____X___P2____X___P3____ ...  X___Pn_____XXX
                 _________
  tuser   XXXXXX__01(SOF)X___00____X___00____X___00____ ...  X_10(EOF)__XXX
                 ^                                             ^
                 SOF=1: フレーム最初のピクセル                 EOF=1: フレーム最後のピクセル

  tuser encoding:  01 = SOF (bit[0]=1, bit[1]=0)
                   10 = EOF (bit[0]=0, bit[1]=1)
                   00 = 通常データ
```

- SOF はフレームの最初のピクセル転送時に `tuser[0] = 1` としてアサートされる。
- EOF はフレームの最後のピクセル転送時に `tuser[1] = 1` としてアサートされる。
- 中間のピクセルでは `tuser = 2'b00` となる。

### 5.4 ブロック境界 (tlast) - DCT 出力

DCT モジュール出力における 8x8 ブロック境界の `tlast` シグナリングを示す。

```
             ___     ___     ___     ___     ___     ___     ___     ___
  aclk    __|   |___|   |___|   |___|   |___|   |___|   |___|   |___|   |__
                 _______________________________________________ ___________
  tvalid  ______|                                               |
                 _______________________________________________ ___________
  tready  ______|                                               |
                 _________ _________ _________   _________ _________ ______
  tdata   XXXXXX__C[0]___X__C[1]___X__C[2]___ ...C[62]___X__C[63]__X__C[0]_
                                                           _________
  tlast   __________________________________________________|       |_______
                 _________ _________ _________   _________ _________ ______
  tuser   XXXXXX___0100__X___0000__X___0000___ ...___0000__X___0000__X_0001_
                 (SOF,Y)                                             (Cb)

  comp_id:  00=Y ブロック → 01=Cb ブロック → 10=Cr ブロック

  |<------------ 64 coefficients (1 block) ----------->|<-- next block -->
```

- C[0]~C[63]: 1 ブロック (8x8=64) の DCT 係数。
- `tlast` は各ブロックの最後の係数 (index 63) で High になる。
- `tuser` の `comp_id` フィールドでコンポーネント (Y/Cb/Cr) を識別する。
- SOF はフレーム最初のブロックの最初の係数で示される。

## 6. データフロー一覧

パイプライン各ステージ間のデータフォーマットを一覧する。

| Stage | 接続 (Source → Dest)              | tdata 幅 | tdata フォーマット              | tuser 幅 | tuser フォーマット               | tlast の意味         |
|-------|----------------------------------|-----------|-------------------------------|-----------|----------------------------------|----------------------|
| 0     | Input → rgb2ycbcr                | 32        | A8R8G8B8                       | 2         | {EOF, SOF}                       | 行末 (End of Row)    |
| 1     | rgb2ycbcr → dct_2d               | 24        | {Y[23:16], Cb[15:8], Cr[7:0]} | 2         | {EOF, SOF}                       | 行末 (End of Row)    |
| 2     | dct_2d → quantizer               | 12        | signed DCT coefficient         | 4         | {EOF, SOF, comp_id[1:0]}        | ブロック末尾 (idx 63)|
| 3     | quantizer → zigzag_scan          | 12        | signed quantized coefficient   | 4         | {EOF, SOF, comp_id[1:0]}        | ブロック末尾 (idx 63)|
| 4     | zigzag_scan → rle_encoder        | 12        | zigzag-ordered coefficient     | 4         | {EOF, SOF, comp_id[1:0]}        | ブロック末尾 (idx 63)|
| 5     | rle_encoder → huffman_encoder    | 24        | rle_symbol_t                   | 4         | {EOF, SOF, comp_id[1:0]}        | ブロック末尾 (symbols)|
| 6     | huffman_encoder → bitstream_asm  | 32        | MSB-aligned codeword           | 7         | {EOF, SOF, code_length[4:0]}    | ブロック末尾 (codes) |
| 7     | bitstream_asm → Output           | 8         | JPEG byte                      | 2         | {EOF, SOF}                       | JPEG ファイル末尾    |

### tkeep 使用状況

| Stage | tkeep 幅 | 説明                                               |
|-------|----------|----------------------------------------------------|
| 0     | 4        | 入力ピクセルのバイトイネーブル (通常 4'b1111)       |
| 1-6   | N/A      | 未使用 (すべての転送が有効)                        |
| 7     | 1        | 出力バイト有効 (通常 1'b1)                          |

## 7. 設計制約・備考

### 7.1 クロックおよびリセット

| 項目       | 仕様                                              |
|------------|---------------------------------------------------|
| クロック   | 単一クロックドメイン (`aclk`)                      |
| 動作周波数 | 100-200 MHz (ターゲットデバイス依存)              |
| リセット   | 非同期 Active-Low リセット (`aresetn`)             |
| リセット幅 | 最低 2 クロックサイクル                           |

### 7.2 画像制約

| 項目           | 仕様                                            |
|----------------|--------------------------------------------------|
| 最大画像幅     | 4096 ピクセル                                    |
| 最大画像高さ   | 4096 ピクセル                                    |
| 画像幅制約     | 8 の倍数であること (非倍数の場合はパディング必要) |
| 画像高さ制約   | 8 の倍数であること (非倍数の場合はパディング必要) |
| 最小画像サイズ | 8x8 ピクセル                                     |
| ピクセルフォーマット | A8R8G8B8 (32-bit, Alpha 未使用)            |

### 7.3 処理仕様

| 項目                 | 仕様                                           |
|----------------------|-------------------------------------------------|
| JPEG プロファイル    | Baseline (8-bit DCT, Sequential, Huffman)       |
| クロマサブサンプリング | 4:4:4 (サブサンプリングなし)                 |
| MCU 構成             | 1Y + 1Cb + 1Cr = 3 ブロック/MCU                |
| DCT 精度             | 8-bit 入力, 12-bit 係数出力                     |
| 量子化テーブル       | JPEG 標準テーブル (Annex K)                     |
| ハフマンテーブル     | JPEG 標準テーブル (Annex K)                     |

### 7.4 スループットターゲット

| 項目                   | 仕様                                         |
|------------------------|-----------------------------------------------|
| 入力スループット       | 1 pixel/clock (sustained)                     |
| パイプラインフル動作時 | 1 pixel/clock スループット維持                |
| バックプレッシャ対応   | 全モジュールで AXI4-Stream バックプレッシャをサポートすること |

### 7.5 バックプレッシャ要件

- すべてのモジュールは AXI4-Stream のバックプレッシャ (`tready`) を正しくサポートしなければならない。
- `tready` が Low の間、Master はデータを廃棄してはならない。
- `tready` が Low の間、Master は `tvalid` と `tdata` を保持しなければならない。
- パイプラインのどのステージでバックプレッシャが発生しても、上流モジュールに適切に伝搬しなければならない。

### 7.6 共通パッケージ

- すべてのモジュールは `jpeg_encoder_pkg.sv` に定義された共通の型定数を使用すること。
- 共通パッケージには以下が含まれる:
  - データ型定義 (`rle_symbol_t` 等)
  - 量子化テーブル定数
  - ハフマンテーブル定数
  - ジグザグオーダーテーブル
  - 画像パラメータ上限値
  - 共通パラメータおよびローカルパラメータ

### 7.7 設計上の注意事項

1. **パイプライン整合性**: `tuser` および `tlast` はデータパスと同じパイプラインレイテンシで遅延させること。`tdata` と `tuser`/`tlast` 間のスキューは許容されない。
2. **リセット後の初期化**: リセット解除後、すべての出力 `tvalid` は Low を保持し、最初の有効データを受信するまで出力しないこと。
3. **DC 値レジスタ初期化**: `rle_encoder` の DC 差分レジスタは SOF 受信時にゼロクリアすること。フレーム間でのリセット忘れに注意すること。
4. **パディング処理**: 画像サイズが 8 の倍数でない場合、`dct_2d` モジュール内で境界ブロックのパディング (ゼロパディングまたは端ピクセル複製) を行うこと。
5. **バイトスタッフィング**: `bitstream_assembler` は `0xFF` データバイト後のスタッフバイト `0x00` 挿入を確実に実装すること。スタッフバイト挿入中は上流へのバックプレッシャが発生する。
6. **ヘッダ生成**: `bitstream_assembler` は SOF 検出時にヘッダを生成する。ヘッダ出力完了までエントロピー符号化データの入力を受け付けない (バックプレッシャで制御)。
